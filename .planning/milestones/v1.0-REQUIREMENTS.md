# Requirements: PoseEngine

**Defined:** 2026-05-01
**Core Value:** The engine must make it trivially easy to add a new pose-based game by implementing a well-defined plugin interface — without touching the engine core or understanding its internals.

## v1 Requirements

### Engine Core

- [x] **ENG-01**: WebSocket server exposes `/ws/player/{room_code}` and `/ws/spectator/{room_code}` endpoints using Axum + Tokio
- [x] **ENG-02**: Room registry uses DashMap with 6-char alphanumeric code generation; rooms created on demand
- [x] **ENG-03**: Each room runs as an independent Tokio task (actor model) exclusively owning all per-room state; no cross-room shared mutexes
- [x] **ENG-04**: 60Hz game loop driven by `tokio::time::interval` with `MissedTickBehavior::Skip` inside the room actor's `select!` loop
- [x] **ENG-05**: Each player connection has a dedicated outbound Tokio task with a bounded `mpsc` channel; game loop never calls `ws.send().await` directly
- [x] **ENG-06**: RTT input delay buffer (fairness cutoff) ported from Python `input_delay.py`; plugin receives only already-released frames in each tick
- [x] **ENG-07**: Pose data fan-out: `MsgPoseUpdate` broadcast to spectators immediately on frame arrival, independent of game loop tick rate
- [x] **ENG-08**: Two broadcast channels per room — fast path for pose updates, slow path for game state and lifecycle events
- [x] **ENG-09**: 3.8-second warmup window zeroes the input buffer; plugin receives empty frame slices during warmup
- [x] **ENG-10**: Round lifecycle managed by engine: on `RoundOver` event from plugin, engine broadcasts `MsgRoundEnd`, increments win counter, calls `on_round_reset`
- [x] **ENG-11**: Calibration handshake: engine calls `on_calibration_complete(slot, reference_velocity)` after both players submit; game loop starts once both are calibrated
- [x] **ENG-12**: Static file serving of `mobile/dist` at `/mobile` and `overlay/dist` at `/overlay` (same paths as Python server)
- [x] **ENG-13**: Joinable Tokio task handles tracked with abort-on-drop guarantee; no zombie game loop tasks after room teardown

### Protocol

- [x] **PROTO-01**: All wire message types modelled in Rust with `serde_json` using `#[serde(tag = "type")]`; field names, discriminators, and optional fields match `shared/protocol.ts` exactly
- [x] **PROTO-02**: Golden-file roundtrip tests: each message type serialized in Rust and compared to reference JSON fixtures; suite runs in CI
- [x] **PROTO-03**: `shared/protocol.ts` kept in sync with Rust models; generation script updated or replaced

### Plugin Trait

- [x] **PLUG-01**: `GamePlugin` trait defines: `init_state() -> Box<dyn Any + Send>`, `on_tick(&self, ctx: &mut TickContext, state: &mut dyn Any) -> Vec<GameEvent>`, plus `on_player_join`, `on_player_leave`, `on_calibration_complete`, `on_round_reset` with default no-op implementations
- [x] **PLUG-02**: `TickContext` struct carries: released pose frames per player slot, `TickInfo { tick, elapsed_secs, remaining_secs }`, `RoomView` (read-only player count/slot state), and helper methods for broadcasting and sending to individual players
- [x] **PLUG-03**: `GameEvent` enum covers at minimum: `Hit { attacker, defender, region, damage, position }`, `RoundOver { winner: Option<u8> }`, `SendToPlayer { slot, payload }`, `CommentaryHint { kind, payload }`
- [x] **PLUG-04**: Plugin state stored as `Box<dyn Any + Send>` per room in the engine; each plugin downcasts it in its own methods — engine never inspects plugin state
- [x] **PLUG-05**: Trait is fully object-safe (`Box<dyn GamePlugin + Send>` usable); all methods synchronous (no `async fn`)
- [x] **PLUG-06**: Engine delivers pose keypoints normalized to hip-centred Y-up coordinates before passing to plugin; plugin never calls coordinate transforms

### Boxing Plugin

- [x] **BOX-01**: Punch detection: wrist speed exceeds calibrated threshold over a 10-frame sliding window; direction and height classify as valid punch
- [x] **BOX-02**: Kick detection: ankle elevation + speed exceeds calibrated threshold
- [x] **BOX-03**: Body region classification: 9 regions (head_chin, head_face, head_throat, torso_upper, torso_lower, block_hand, block_forearm, leg_thigh, leg_shin)
- [x] **BOX-04**: Guard blocking: defender with wrists above guard threshold blocks head/upper-torso hits
- [x] **BOX-05**: Velocity-scaled damage: damage scales linearly with attacker velocity relative to calibrated `reference_velocity`
- [x] **BOX-06**: HP tracking: 800 HP per player; round over when HP reaches 0 (KO) or 90-second timer expires (decision by HP)
- [x] **BOX-07**: Hit cooldown: 12-tick (200ms) per-attacker lockout after each hit to prevent double-counting
- [x] **BOX-08**: Round draw handling: equal HP at time expiry produces a draw outcome
- [x] **BOX-09**: Multi-round match: engine-tracked win counter; match ends when a player reaches `max_wins`
- [x] **BOX-10**: Solo/bot mode: server-injected static pose for P2 slot with scripted random hit timer; three difficulty tiers (easy/normal/hard)

### Bug Fixes

- [x] **FIX-01**: Calibration persists through rematch — `reference_velocity` stored on `PlayerSlot` for the Room lifetime; `on_round_reset` clears only round-scoped state (HP, cooldown counters, combo trackers)
- [x] **FIX-02**: Spectator reconnect state restoration — on spectator WebSocket connect, engine sends a snapshot of current HP, wins, round number, and elapsed time before the client enters the live broadcast stream

### Reference Game

- [x] **GAME2-02**: Second game requires zero changes to engine code; any trait additions needed to support it are considered interface bugs

## Out of Scope

| Feature | Reason |
|---------|--------|
| WASM / dynamic plugin loading | Compile-time native Rust is sufficient; runtime plugin loading adds significant complexity |
| Async plugin methods | Breaks object safety; `async-trait` allocates per call at 60Hz; all side effects via returned `GameEvent` |
| Plugin-visible WebSocket types | Transport layer internals stay in engine; plugin uses `send_to_player` / `broadcast` helpers |
| User accounts / authentication | 6-char room code model retained; security model unchanged from Python |
| Horizontal scaling / room sharding | Single-process Tokio is sufficient for current use case |
| Persistent state (match history, ratings) | No external store; server restart clears all rooms |
| Browser-based game IDE | Deferred until AI generation is tackled |
| TypeScript client changes | Mobile and overlay apps are unchanged; engine must speak existing wire protocol |
| Python server kept running in parallel | Full cutover on completion; no hybrid deployment |
