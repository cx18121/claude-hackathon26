## Summary

Phase 10 adds two new Rust crates to the engine workspace (`boxing-core` and `fps-boxing-plugin`) and registers `FPSBoxingPlugin` in engine-core's plugin HashMap. The dominant work is a crate extraction: `hit_detection.rs` and `damage.rs` move from `boxing-plugin/src/` to `boxing-core/src/`, then both `boxing-plugin` and `fps-boxing-plugin` declare a path dependency on `boxing-core`.

The plugin implementation itself follows an established pattern — `FPSBoxingPlugin` is structurally a simplified `BoxingPlugin`: same `Box<dyn Any + Send>` state pattern, same `on_tick` returns `Vec<GameEvent>`, same `on_calibration_complete`/`on_round_reset` structure. The critical difference is the per-tick `GameEvent::SendToPlayer` broadcasts: each player receives a `MsgFpsState` with their opponent's 6 arm landmarks, and a `MsgFpsHit` is sent to the player who was hit.

Protocol struct additions in `protocol.rs` follow a mechanical pattern — `#[derive(Serialize, Deserialize, TS)]` + `#[ts(export)]` + a `default_type_*` function. TypeScript output is regenerated by `cargo test` in the engine workspace, which triggers ts-rs exports to `../../shared/` (configured in `engine/.cargo/config.toml`).

**Primary recommendation:** Implement in three sequential units: (1) create boxing-core crate and update boxing-plugin, (2) implement fps-boxing-plugin, (3) wire up protocol.rs + main.rs + regenerate TS.

## Common Pitfalls

### Pitfall 1: Forgetting to re-export boxing-core items in boxing-plugin after extraction
**What goes wrong:** After moving `hit_detection.rs` and `damage.rs` to boxing-core, `boxing-plugin/src/lib.rs` still has `mod hit_detection; mod damage;` — these will fail to compile because the files no longer exist in that crate.
**Why it happens:** Mechanical file move without updating the declaring crate.
**How to avoid:** After moving the files, change boxing-plugin/src/lib.rs to `use boxing_core::hit_detection; use boxing_core::damage;` (or inline the call sites directly).
**Warning signs:** `error[E0583]: file not found for module` at compile time.

### Pitfall 2: Building the plugin registry after Arc::new
**What goes wrong:** `Arc::new(AppState { ... })` freezes the HashMap — you cannot `.insert()` after wrapping. The existing code comment in main.rs line ~401 already calls this out: "Build plugin registry before Arc::new — cannot insert after wrapping (Pitfall 2)".
**Why it happens:** `Arc<T>` is immutable; `HashMap::insert` requires `&mut`.
**How to avoid:** Insert `"fps_boxing"` into the plugins HashMap before `Arc::new(AppState { ... })`. Follow the exact same pattern as the existing boxing/dance insertions (main.rs lines 401-403).
**Warning signs:** Borrow checker error `cannot borrow as mutable` on the Arc.

### Pitfall 3: Sending MsgFpsState as Broadcast instead of SendToPlayer (per-player)
**What goes wrong:** Using `GameEvent::Broadcast` sends to all connected clients including spectators. Each player needs their opponent's landmarks — Player A needs Player B's arms, Player B needs Player A's arms. They are different messages.
**Why it happens:** Broadcast is simpler — single event. But it would send the wrong landmarks to each player.
**How to avoid:** Emit two `GameEvent::SendToPlayer` events per tick: one to slot 1 carrying slot 2's landmarks, one to slot 2 carrying slot 1's landmarks. See dispatch_events in game_loop.rs for how SendToPlayer is dispatched (line ~219).
**Warning signs:** Both players see the same landmark data; first-person arms mirror incorrectly.

### Pitfall 4: Accessing opponent landmarks when frames are empty
**What goes wrong:** `ctx.frames[opponent_idx].back()` returns `None` if the opponent hasn't sent any frames yet (they are lagging or not yet connected). Unwrapping it panics.
**Why it happens:** Frames are delivered from an RTT-gated buffer — may be empty on any given tick.
**How to avoid:** Use `.and_then(|f| f.keypoints.get(idx).cloned())` with a fallback (e.g., `PoseKeypoint { x: 0.0, y: 0.0, z: 0.0, visibility: 0.0 }`). Match the existing `detect_punch` pattern which returns early if `attacker_frames.len() < 3`.
**Warning signs:** Occasional panics in production under high latency.

### Pitfall 5: Wrong MediaPipe indices for the 6 arm landmarks
**What goes wrong:** Using incorrect keypoint indices produces garbage landmark positions. D-07 specifies MediaPipe indices 11–16.
**Why it happens:** MediaPipe has 33 landmarks; easy to confuse elbow/wrist indices.
**How to avoid:** The constants already exist in `boxing-core/src/hit_detection.rs` (moving from boxing-plugin). Reference them:
- `LEFT_SHOULDER = 11`, `RIGHT_SHOULDER = 12`
- `LEFT_ELBOW = 13`, `RIGHT_ELBOW = 14`  ← not yet constants in hit_detection.rs, need to add
- `LEFT_WRIST = 15`, `RIGHT_WRIST = 16`
**Critical gap:** `hit_detection.rs` currently defines `WRIST_LEFT = 15`, `WRIST_RIGHT = 16`, `LEFT_SHOULDER = 11`, `RIGHT_SHOULDER = 12` but does NOT define elbow constants (13, 14). FPSBoxingPlugin needs to define `LEFT_ELBOW: usize = 13` and `RIGHT_ELBOW: usize = 14` locally in its lib.rs (or add them to boxing-core/src/hit_detection.rs as pub consts).
**Warning signs:** Elbow positions are clearly wrong in the FPS view.

### Pitfall 6: gen_protocol.py is DEPRECATED — do not use it
**What goes wrong:** Running `scripts/gen_protocol.py` exits with an error (by design) and overwrites nothing. It was replaced in Phase 1 by ts-rs.
**Why it happens:** CONTEXT.md section D-04 references the script by name — this is stale documentation from the discussion phase.
**How to avoid:** Regenerate `shared/protocol.ts` by running `cargo test` from the `engine/` directory. ts-rs exports all `#[ts(export)]` structs to `../../shared/` (configured in `engine/.cargo/config.toml` via `TS_RS_EXPORT_DIR`). The script explicitly prints: "Run `cargo test` from engine/ to regenerate shared/protocol.ts via ts-rs."

### Pitfall 7: MsgFpsState uses PoseKeypoint from protocol.rs, not plugin_trait
**What goes wrong:** There are TWO `PoseKeypoint` types in this codebase: `plugin_trait::PoseKeypoint` (used by plugins, no Serialize) and `engine_core::protocol::PoseKeypoint` (has `#[derive(Serialize, Deserialize, TS)]`). Protocol message structs must use the one from protocol.rs.
**Why it happens:** Both types have the same fields (x, y, z, visibility f64) but different traits.
**How to avoid:** `MsgFpsState` in `protocol.rs` uses `crate::protocol::PoseKeypoint` (the local type with Serialize). In FPSBoxingPlugin's `on_tick`, convert from `plugin_trait::PoseKeypoint` to `serde_json::Value` using `json!({ "x": kp.x, "y": kp.y, "z": kp.z, "visibility": kp.visibility })` OR define the payload directly as a JSON value. Alternatively, define `MsgFpsState` with `PoseKeypoint` fields that map to the protocol type and serialize with `serde_json::to_value`.
**Concrete pattern:** See how `MsgYouWereHit` uses plain JSON in boxing-plugin (`json!({ "type": "you_were_hit", "region": ..., "damage": ... })`) vs. the typed struct in protocol.rs. Both approaches work; the typed struct approach (serialize to Value before putting in SendToPlayer) matches the dance plugin pattern.

### how build_fps_state extracts 6 arm keypoints from a PoseFrame
```rust
// Landmark indices (same as hit_detection.rs constants; elbow not yet a const there)
const LEFT_SHOULDER:  usize = 11;
const RIGHT_SHOULDER: usize = 12;
const LEFT_ELBOW:     usize = 13;
const RIGHT_ELBOW:    usize = 14;
const LEFT_WRIST:     usize = 15;
const RIGHT_WRIST:    usize = 16;

fn zero_kp() -> protocol::PoseKeypoint {
    protocol::PoseKeypoint { x: 0.0, y: 0.0, z: 0.0, visibility: 0.0 }
}

fn extract_kp(frame: &plugin_trait::PoseFrame, idx: usize) -> protocol::PoseKeypoint {
    frame.keypoints.get(idx)
        .map(|kp| protocol::PoseKeypoint { x: kp.x, y: kp.y, z: kp.z, visibility: kp.visibility })
        .unwrap_or_else(zero_kp)
}
```

## Punch Type String Derivation

D-06 says `punch_type` uses same string values as existing boxing protocol (e.g. `"jab"`, `"cross"`, `"hook"`). The existing boxing protocol does NOT define punch_type — `MsgYouWereHit` has `region: String`, not punch type. The bot.rs uses `"bot_punch"` as a region string. There is no canonical punch_type enum in the current codebase.

**Recommendation:** Map `BodyRegion` to punch_type string in FPSBoxingPlugin using a simple local function:
```rust
fn region_to_punch_type(region: &plugin_trait::BodyRegion) -> String {
    match region {
        BodyRegion::HeadFace | BodyRegion::HeadChin | BodyRegion::HeadThroat => "cross",
        BodyRegion::TorsoUpper | BodyRegion::TorsoLower => "body_shot",
        BodyRegion::LegThigh | BodyRegion::LegShin => "kick",
        BodyRegion::BlockHand | BodyRegion::BlockForearm => "blocked",
    }.to_string()
}
```
This is a Phase 10 placeholder — Phase 14 can refine based on which wrist struck (left = jab, right = cross) if the client needs that distinction. Since D-06 says `punch_type: String` (not an enum), this is forward-compatible.
[ASSUMED: The mapping above is a reasonable default. The CONTEXT.md does not specify which BodyRegion maps to which punch type string, only that "jab/cross/hook" are examples.]

## Security Domain

This phase is a server-side Rust plugin with no authentication, no file I/O, no SQL, and no new HTTP endpoints. All network communication goes through the existing engine-core WebSocket handler.

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|

### Known Threat Patterns
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Out-of-bounds keypoint access | Tampering/DoS | `.get(idx).unwrap_or(zero_kp)` — all landmark extractions |

| Slot index out of bounds in SendToPlayer | Tampering | slot value from plugin (1 or 2) — checked by dispatch_events at game_loop.rs line 222 |

## Open Questions

1. **Where to put elbow constants (indices 13, 14)**
   - What we know: `hit_detection.rs` defines shoulder (11, 12) and wrist (15, 16) constants but not elbows. FPSBoxingPlugin needs indices 13 and 14.
   - What's unclear: Should they be added as `pub const` to boxing-core/src/hit_detection.rs (making them available to any consumer) or defined locally in fps-boxing-plugin/src/lib.rs?
   - Recommendation: Add `pub const LEFT_ELBOW: usize = 13; pub const RIGHT_ELBOW: usize = 14;` to boxing-core/src/hit_detection.rs alongside the existing shoulder/wrist constants. This keeps all MediaPipe index constants in one place.

2. **hp field type in MsgFpsState: (u32, u32) vs struct**
   - What we know: D-05 specifies `hp: (u32, u32)`. `MsgRoundEnd` also uses `final_hp: (u32, u32)` as a tuple at protocol.rs line 193. MsgGameState uses `hp: (u32, u32)` at line 265.
   - What's unclear: ts-rs renders Rust tuples as TypeScript tuples `[number, number]` — this is consistent with the existing protocol. No ambiguity.
   - Recommendation: Use `(u32, u32)` tuple to match existing protocol.rs conventions.
